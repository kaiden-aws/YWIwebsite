  <action>
Step 1 — Build locally to catch errors before deploying:
```bash
npm run build
```
If build fails, diagnose and fix before proceeding.

Step 2 — Deploy to Vercel preview:
```bash
npx vercel --yes
```
Capture the preview URL from output. If authentication is needed, pause and create a checkpoint for the user to authenticate via `npx vercel login`.

Step 3 — Wait ~30 seconds for the deployment to be fully ready, then run Lighthouse CLI audits against all 6 pages. Run each audit in desktop mode with JSON output:

Pages to audit:
- `/` (homepage)
- `/about`
- `/services`
- `/products`
- `/gallery`
- `/contact`

For each page, run:
```bash
npx lighthouse {PREVIEW_URL}{path} --output=json --output-path=./lighthouse-{page}.json --chrome-flags="--headless --no-sandbox" --only-categories=performance,accessibility,best-practices,seo
```

Step 4 — Parse the JSON results for each page. Extract the four category scores (multiply by 100 to get the 0-100 scale). Create a summary table of all scores.

Step 5 — Evaluate results:
- If ALL scores across ALL pages are 90+: Phase passes. Record scores.
- If ANY score is below 90: Analyze the Lighthouse audit details for that page/category. Identify the specific failing audits. Attempt fixes for code-level issues (e.g., missing alt text, render-blocking resources, missing meta tags). Re-deploy and re-audit affected pages after fixes.

Note: Some Lighthouse metrics are network/server dependent and may fluctuate on preview deployments. If a score is 85-89 and caused by server latency rather than code issues, document this but do not consider it a blocking failure — Vercel preview environments have cold-start latency that won't affect production.

Step 6 — Clean up lighthouse JSON files after recording results:
```bash
rm -f lighthouse-*.json
```
