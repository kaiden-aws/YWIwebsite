  <files>

  <action>
    Plan 01 already set up MotionConfig reducedMotion="user" (handles all Framer Motion m.* components) and added `@media (prefers-reduced-motion: reduce) { .animate-bounce { animation: none; } }` to globals.css. This task covers the remaining CSS animations and JS smooth scroll that MotionConfig does not handle.

    **ScrollIndicator.tsx**: Add the `motion-reduce:animate-none` Tailwind class to the element that has `animate-bounce`. This provides a component-level safeguard in addition to the global CSS rule from Plan 01. The element should have both: `animate-bounce motion-reduce:animate-none`.

    **ServicesPreview.tsx**: The service card Link has `hover:-translate-y-2` and the arrow span has `group-hover:translate-x-1`. Add `motion-reduce:hover:translate-y-0` to the Link and `motion-reduce:group-hover:translate-x-0` to the arrow span. These are CSS transitions that MotionConfig does not cover. Keep the `hover:shadow-lg` effect -- shadow changes are safe for reduced motion since they are not transform-based movement.

    **BackToTop.tsx**: The `scrollToTop` function (or equivalent) currently uses `window.scrollTo({ top: 0, behavior: 'smooth' })`. Add a check: `const prefersReducedMotion = window.matchMedia('(prefers-reduced-motion: reduce)').matches` and use `behavior: prefersReducedMotion ? 'auto' : 'smooth'`. This ensures the back-to-top action is instant when reduced motion is preferred. Also note: the Framer Motion enter/exit animation on BackToTop (opacity + scale) is already handled by MotionConfig from Plan 01.
  </action>

    Run `npx next build` -- no build errors. Grep ScrollIndicator.tsx for `motion-reduce`. Grep ServicesPreview.tsx for `motion-reduce`. Grep BackToTop.tsx for `prefers-reduced-motion`.
  </verify>
  <done>
    ScrollIndicator bounce stops when reduced motion is on. ServicesPreview cards do not lift on hover and arrow does not translate when reduced motion is on. BackToTop scrolls instantly (no smooth scroll) when reduced motion is on.
