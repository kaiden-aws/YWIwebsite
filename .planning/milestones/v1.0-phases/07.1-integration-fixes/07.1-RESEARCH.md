# Phase 7.1: Integration Fixes - Research

**Researched:** 2026-02-19
**Domain:** Cross-phase integration quality (Next.js client directives, data source consistency, navigation flows)
**Confidence:** HIGH

## Summary

This phase addresses three discrete integration defects surfaced by the v1.0 milestone audit. All three are straightforward fixes requiring no new libraries, no architectural changes, and no new components. The defects are:

1. **Missing `'use client'` directive in GalleryLightbox.tsx** -- The file uses React hooks (`useEffect`, `useRef`, `useCallback`) and Framer Motion `m` components but lacks the `'use client'` directive. It currently works because GalleryGrid.tsx (which does have `'use client'`) imports it, making it part of GalleryGrid's client boundary. However, this is fragile: if GalleryLightbox were ever imported from a Server Component, it would fail at runtime. Adding the directive makes the component self-documenting and resilient.

2. **Hardcoded services list in Footer.tsx** -- The Footer has a local `services` array with six strings that are manually typed rather than derived from the canonical `services.ts` data file. One label is already wrong: the Footer says `'Irrigation & Lighting'` but the canonical service title in `services.ts` is `'Irrigation & Landscape Lighting'`. The fix is to import from `services.ts` and derive the labels.

3. **Missing "View All Products" CTA in ProductsBanner** -- The homepage ProductsBanner section displays the 7 product categories and the retail yard address but provides no link to the `/products` page. A user viewing the homepage products section has no way to navigate to the full Products page (with the Material Calculator) except via the nav bar. This breaks the intended "Homepage -> Products -> Calculator -> Contact" user flow.

**Primary recommendation:** Apply all three fixes in a single plan. Each fix is isolated (different files), non-conflicting, and independently verifiable.

## Architecture Patterns

### Pattern 1: Self-Documenting Client Directives

**What:** Every component that uses React hooks or browser APIs gets its own `'use client'` directive, even if it is currently only imported by another client component.

**When to use:** Any `.tsx` file that calls `useState`, `useEffect`, `useRef`, `useCallback`, or uses Framer Motion's `m` components.

**Why:** A component without `'use client'` that uses hooks will work fine when imported from a client component (the client boundary propagates down). But if someone later imports it from a Server Component, it will fail at runtime with a cryptic error. The directive costs nothing and makes the contract explicit.

**Current project pattern:** `HeroParallax.tsx`, `TestimonialCarousel.tsx`, `MaterialCalculator.tsx`, and `GalleryGrid.tsx` all follow this pattern. `GalleryLightbox.tsx` is the only exception.

**Example:**
```typescript
// GalleryLightbox.tsx - BEFORE (fragile)
import { useEffect, useRef, useCallback } from 'react'
import { AnimatePresence, m } from 'motion/react'
// ...

// GalleryLightbox.tsx - AFTER (resilient)
'use client'

import { useEffect, useRef, useCallback } from 'react'
import { AnimatePresence, m } from 'motion/react'
// ...
```

### Pattern 2: Single Source of Truth for Data

**What:** UI components derive display labels from canonical data files rather than hardcoding strings.

**When to use:** Anytime a component displays data that also exists in a `src/lib/data/*.ts` file.

**Current violation:** Footer.tsx has a local `const services = [...]` with 6 hardcoded strings. The canonical data is in `src/lib/data/services.ts` which exports `services: Service[]` with a `title` property on each item.

**Example:**
```typescript
// Footer.tsx - BEFORE (hardcoded, already drifted)
const services = [
  'Landscape Design & Build',
  'Residential Maintenance',
  'Commercial Landscaping',
  'Municipal Projects',
  'Irrigation & Lighting',        // WRONG: canonical is "Irrigation & Landscape Lighting"
  'Snow Removal',
]

// Footer.tsx - AFTER (imports from canonical source)
import { services } from '@/lib/data/services'
// Then in JSX: services.map((s) => s.title)
```

**Important consideration:** Footer.tsx is currently a pure Server Component (no `'use client'` directive). Importing `services.ts` keeps it as a Server Component because `services.ts` is also a pure data module with no client-side APIs. This is the correct approach -- no `'use client'` is needed on Footer.

### Pattern 3: CTA Links for User Flow Continuity

**What:** Each homepage section that previews content from another page must include a clear CTA linking to the full page.

**Current pattern in other sections:**
- `ServicesPreview` links to `/services` via service card CTAs
- `AboutTeaser` links to `/about` via "Learn More" button
- `ProjectShowcase` links to `/gallery` via image clicks

**Missing:** ProductsBanner has no link to `/products`. The "Visit our retail yard" text with the address is not a link -- it is purely informational.

**Example:**
```typescript
// Add after the retail yard address div, within the text-center mt-12 container
import Link from 'next/link'

<Link
  href="/products"
  className="inline-block mt-6 px-8 py-3 bg-terracotta text-cream rounded-lg font-medium hover:bg-terracotta/90 transition-colors"
>
  View All Products
</Link>
```

### Anti-Patterns to Avoid

- **Renaming the import variable:** The Footer currently uses `const services = [...]` as a local constant. When switching to the import from `services.ts`, be aware that `services.ts` also exports `services`. If there is a naming conflict with the existing import from `navigation.ts`, rename the local reference (e.g., `import { services as serviceList }`). However, since Footer currently has no `services` import, there is no conflict -- the local `const services` is simply replaced by the import.
- **Adding unnecessary `'use client'` to Footer:** Importing from `services.ts` does NOT require `'use client'` on Footer. The data file exports plain objects and the Footer remains a Server Component.
- **Using a button instead of Link for CTA:** The "View All Products" CTA navigates to another route. Use Next.js `Link`, not a `<button>`, to ensure client-side navigation and prefetching.

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Navigation between pages | `<a>` tags or `onClick` with `router.push` | Next.js `<Link>` component | Client-side navigation, prefetching, active state detection |

## Common Pitfalls

### Pitfall 1: Footer Becoming a Client Component

**What goes wrong:** Adding `'use client'` to Footer.tsx when importing from services.ts, thinking data imports require it.
**Why it happens:** Misunderstanding of the client boundary -- only files that use React hooks or browser APIs need the directive.
**How to avoid:** `services.ts` exports plain objects. It has no hooks, no `useState`, no browser APIs. Importing it in a Server Component is perfectly fine.
**Warning signs:** If Footer gains `'use client'`, it adds unnecessary client-side JavaScript for a component that is purely presentational.

### Pitfall 2: Link vs Button Semantics

**What goes wrong:** Using a `<button onClick={() => router.push('/products')}>` for the CTA, which requires `'use client'` on ProductsBanner.
**Why it happens:** Defaulting to button patterns from React SPA habits.
**How to avoid:** ProductsBanner is currently a Server Component. Use `<Link href="/products">` which works in Server Components and provides client-side navigation. Import `Link` from `next/link`.
**Warning signs:** If ProductsBanner gains `'use client'`, the fix has introduced unnecessary complexity.

### Pitfall 3: Service Label Mismatch After Import

**What goes wrong:** Importing services but still referencing the wrong property (e.g., `s.description` instead of `s.title`).
**Why it happens:** Not checking the `Service` interface shape.
**How to avoid:** The `Service` interface has `title: string` -- use `service.title` to get the display label.
**Warning signs:** Footer service labels look different from the Services page after the fix.

### Pitfall 4: CTA Styling Inconsistency

**What goes wrong:** The "View All Products" button looks different from other CTAs on the homepage.
**Why it happens:** Ad hoc styling instead of matching existing CTA patterns.
**How to avoid:** Match the terracotta CTA button style used elsewhere on the homepage (e.g., hero CTA, services CTA). Use classes like `bg-terracotta text-cream rounded-lg font-medium hover:bg-terracotta/90 transition-colors` consistent with existing buttons.
**Warning signs:** The CTA looks visually out of place compared to other homepage buttons.
